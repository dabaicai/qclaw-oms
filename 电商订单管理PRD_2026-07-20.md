## 一、背景与问题（Background & Problems）

### 1.1 业务背景
平台同时承载 B2C（品牌/自营）与 C2C（个人/小 B 商家）两种交易形态。随着 SKU 规模、订单量、入驻商家数增长，订单系统面临：
- 多角色（买家、平台、商家、仓配、客服、财务）协作复杂；
- 交易形态差异大（一口价、拍卖、拼团、预售、虚拟商品）；
- 售后场景多样（仅退款、退货退款、换货、维修）；
- 合规要求提升（7 天无理由、消保、发票、履约时效）。

### 1.2 当前痛点
1. **订单状态混乱**：缺乏统一状态机，各端（买家端/商家端/后台）展示口径不一致，易引发客诉。
2. **履约链路断点**：支付成功但未通知仓储、发货后物流回传延迟，订单"卡单"难定位。
3. **售后无标准流程**：退款/退货规则散落在人工判断，时效与责任不清，纠纷率高。
4. **对账困难**：平台、商家、支付渠道三方金额不一致，结算周期长、差错靠人工捞。
5. **查询与导出弱**：大促期间订单检索慢，运营/客服无法灵活筛选与批量处理。
6. **风控缺失**：刷单、超卖、黄牛囤货、异常地址缺乏实时拦截。

### 1.3 目标用户
- **买家（C 端用户）**：关注下单顺畅、物流可见、售后无忧。
- **商家（B 端/C 端卖家）**：关注接单效率、发货履约、退款处理、对账清晰。
- **平台运营/客服**：关注工单处理、纠纷仲裁、数据看板。
- **仓储/物流**：关注拣货、发货、回传。
- **财务/结算**：关注对账、分账、开票。

## 二、目标与价值（Goals & Value）

### 2.1 目标（Goal）
1. **统一订单模型与状态机**，一套模型支撑 B2C/C2C 全场景。
2. **端到端履约可视化**，从下单到签收全程状态可追踪、可干预。
3. **标准化售后闭环**，规则可配置、时效可监控、责任可归属。
4. **自动化对账与结算**，减少人工差错，缩短结算周期。
5. **可观测与可扩展**，支持大促峰值与未来业务形态（跨境、O2O）。

### 2.2 价值（Value）
- 买家：履约时效提升、售后体验提升 → NPS 提升。
- 商家：接单/发货/退款效率提升，对账自动化 → 运营成本下降。
- 平台：纠纷率下降、对账差错率下降、人工客服压力下降。
- 技术：统一模型降低多系统维护成本，状态机可测试、可演进。

### 2.3 成功指标（North Star & Guardrails）
| 指标 | 现状目标 | 目标值 |
| --- | --- | --- |
| 订单创建成功率 | - | ≥ 99.95% |
| 支付→发货平均时效 | - | ≤ 24h（现货） |
| 售后纠纷率 | - | ≤ 1.5% |
| 对账差错率 | - | ≤ 0.1% |
| 订单列表 P95 查询延迟 | - | ≤ 800ms |
| 大促峰值订单吞吐 | - | ≥ 5 万单/分钟 |

## 三、范围（Scope）

### 3.1 In Scope（范围内）
- 订单创建、合并、拆单（按商家/SKU/仓库）。
- 统一订单状态机与状态流转规则。
- 支付集成（多支付渠道、支付结果回调）。
- 发货与物流（电子面单、物流轨迹回传）。
- 售后服务（退款、退货退款、换货；含平台仲裁）。
- 订单管理后台（商家后台 + 平台运营后台）。
- 订单查询、筛选、导出、批量操作。
- 消息通知（短信/站内信/Push/邮件）。
- 对账与结算数据（对账文件 + 分账规则）。
- 基础风控（超卖、重复支付、异常地址）。

### 3.2 Out of Scope（范围外，本期不做）
- 跨境清关、多币种结算（预留扩展点）。
- 营销玩法引擎本身（拼团/秒杀由营销系统承载，订单仅消费结果）。
- 完整 CRM / 会员积分（仅消费用户标识）。
- 仓储 WMS 内部作业（仅对接发货接口与回传）。
- 法务合同与电子签章（仅对接）。

## 四、用户角色与权限（Personas & RBAC）

| 角色 | 简称 | 关键能力 | 数据范围 |
| --- | --- | --- | --- |
| 买家 | Buyer | 下单、支付、查物流、申请售后、评价 | 仅本人订单 |
| 商家 | Merchant | 接单、发货、处理退款、对账 | 本店订单 |
| 平台运营 | Ops | 订单监控、纠纷仲裁、规则配置 | 全平台 |
| 平台客服 | CS | 工单处理、代客操作（有限） | 全平台（受限） |

| 财务 | Fin | 对账、结算、开票 | 全平台（脱敏） |

## 五、核心概念与术语（Glossary）

| 术语 | 说明 |
| --- | --- |
| 订单 Order | 一次交易的唯一凭证，含商品、价格、收货、支付、履约信息 |
| 子订单 SubOrder | 拆单后归属同一商家的履约单元（一个订单可含多个子订单） |
| 订单状态 OrderStatus | 订单生命周期节点（待支付/已支付/待发货…） |
| 支付单 PayOrder | 与订单关联的收款单据，支持多笔支付（优惠券/余额/第三方） |
| 履约 Fulfillment | 从支付到签收的发货、物流过程 |
| 售后单 AfterSale | 退款/退货/换货请求，关联原订单 |
| 对账 Reconciliation | 平台、商家、支付渠道三方金额核对 |
| 分账 Split | 按规则将收款在平台与商家间分配 |
| 超时关单 AutoClose | 规定时间内未支付自动取消并释放库存 |

## 六、整体业务流程（Business Flow）

### 6.1 正向交易主流程
```
[商品详情] → [下单(创建订单草稿)] → [确认/改地址/用券]
   → [发起支付] → [支付渠道] → [支付成功回调]
   → [订单状态: 已支付] → [风控校验]
   → [拆单(按商家/仓库)] → [商家接单]
   → [拣货/出库] → [电子面单/发货] → [物流轨迹回传]
   → [买家签收] → [确认收货/自动确认]
   → [订单完成] → [评价/结算分账]
```

### 6.2 逆向（售后）流程
```
[买家申请售后] → [商家审核]
   ├─ 仅退款(未发货): 平台/商家同意 → 退款 → 关单
   ├─ 退货退款: 买家寄回 → 商家收货验货 → 退款
   ├─ 换货: 买家寄回 → 商家重发 → 新物流
   └─ 拒绝: 买家可发起平台仲裁 → 平台判定 → 执行
```

### 6.3 异常分支
- 支付失败/超时 → 关单释放库存。
- 超时未发货 → 自动提醒/平台介入/赔付。
- 物流长时间无更新 → 预警 + 买家可催单。
- 退款失败（渠道异常）→ 重试 + 人工兜底。

## 七、功能需求（Functional Requirements）

> 优先级：P0 必须首版；P1 首版或紧跟；P2 后续迭代。

### 7.1 下单与订单创建（P0）
- **FR-1.1** 支持从购物车/立即购买创建订单，合并同一商家商品。
- **FR-1.2** 下单时校验：商品上架状态、库存（预占）、收货地址、限购、风控。
- **FR-1.3** 支持优惠叠加计算（店铺券、平台券、积分、余额），金额需可追溯（价格明细）。
- **FR-1.4** 生成订单草稿（待支付），预占库存，超时未支付自动关单释放库存（默认 30 分钟，可配置）。
- **FR-1.5** 支持订单备注、发票信息（纸质/电子）、指定送达时间。
- **FR-1.6** 虚拟商品（卡密/充值）下单后直接出库，不走物流。

### 7.2 支付（P0）
- **FR-2.1** 集成至少 2 个支付渠道（如微信支付、支付宝），支持余额/银行卡。
- **FR-2.2** 支付结果以渠道回调 + 主动查单双保险确认，防止掉单。
- **FR-2.3** 支持部分支付、组合支付；支付超时关闭支付单。
- **FR-2.4** 支付成功后推送订单至履约，触发拆单与商家接单。

### 7.3 订单状态机（P0，详见第八节）
- **FR-3.1** 统一状态枚举与流转规则，所有端以同一状态源为准。
- **FR-3.2** 状态变更需带原因、操作人、时间戳，写入订单流水（OrderLog）。
- **FR-3.3** 非法状态流转拒绝并告警。

### 7.4 拆单与发货（P0）
- **FR-4.1** 按商家强制拆子订单；可按仓库/物流策略二次拆。
- **FR-4.2** 商家端支持单件/批量发货，生成电子面单（对接快递鸟/菜鸟等）。
- **FR-4.3** 发货后回传物流单号，订阅物流轨迹（揽收/在途/派送/签收）。
- **FR-4.4** 支持修改物流、补发、拆分包裹（多包裹发货）。

### 7.5 物流与签收（P0）
- **FR-5.1** 买家端展示实时物流轨迹。
- **FR-5.2** 支持"确认收货"，超时未操作则系统自动确认（默认 7~15 天可配）。
- **FR-5.3** 异常物流（停滞/退回）预警，触发客服跟进。

### 7.6 售后（P0）
- **FR-6.1** 支持仅退款、退货退款、换货三种类型；未发货/已发货/已签收不同可售后期（可配）。
- **FR-6.2** 退款原路返回，支持部分退款、多次退款；退款金额校验（不超实付）。
- **FR-6.3** 退货需填写退回物流，商家验货后退款；超时未处理自动同意（保护买家）。
- **FR-6.4** 纠纷升级平台仲裁，仲裁结果强执行。
- **FR-6.5** 售后全程状态可见，通知双方。

### 7.7 订单管理后台（P0/P1）
- **FR-7.1 商家后台**：订单列表（按状态/时间/关键词筛选）、详情、发货、退款处理、批量导出、备注。
- **FR-7.2 平台运营后台**：全平台订单监控大屏、异常订单池、仲裁台、规则配置（超时/自动确认/可售后期）。
- **FR-7.3 客服台**：工单关联订单，代客操作（发货/退款需审批），操作留痕。
- **FR-7.4 搜索与导出**：按订单号/手机号/商品/时间范围检索；支持 CSV/Excel 导出（脱敏）；大促限流。

### 7.8 消息通知（P1）
- **FR-8.1** 关键节点通知：待支付提醒、支付成功、发货、物流异常、售后进度、签收、退款到账。
- **FR-8.2** 渠道：站内信 + Push + 短信（敏感用短信），支持免打扰与频控。

### 7.9 对账与结算（P1）
- **FR-9.1** 每日生成对账文件（平台 vs 支付渠道 vs 商家）。
- **FR-9.2** 差异自动标记 + 人工复核流程。
- **FR-9.3** 按分账规则生成结算单，支持周期结算（T+1/T+7 可配）。
- **FR-9.4** 发票管理对接（开票申请、电子发票回传）。

### 7.10 风控与防刷（P1/P2）
- **FR-10.1** 下单风控：限购、异常地址、黑名单、设备指纹。
- **FR-10.2** 库存防超卖：预占 + 分布式锁 + 最终一致补偿。
- **FR-10.3** 退款风控：异常高频退款、薅羊毛识别。

### 7.11 数据统计与看板（P2）
- **FR-11.1** 订单量、GMV、转化率、退款率、履约时效多维看板。
- **FR-11.2** 商家履约评分（发货时效、退款纠纷）。

## 八、订单状态机详述（Order State Machine）

### 8.1 订单（主单）状态枚举
| 状态 | 含义 | 进入条件 |
| --- | --- | --- |
| CREATED | 待支付 | 下单成功，生成草稿 |
| PAID | 已支付 | 支付成功回调确认 |
| PARTIAL_SHIPPED | 部分发货 | 部分子订单发货 |
| SHIPPED | 已发货 | 全部子单发货 |
| PARTIAL_RECEIVED | 部分签收 | 部分子单签收 |
| COMPLETED | 已完成 | 全部签收/自动确认 |
| CLOSED | 已关闭 | 超时未付/取消/退款完结 |
| AFTERSALE | 售后中 | 存在进行中售后单 |

> 说明：主单状态为"视图态"，由子单状态与售后单聚合计算，保证单一事实来源。

### 8.2 子订单（子单）状态机
```
CREATED ──支付成功──▶ PAID
PAID ──商家接单──▶ ACCEPTED(可选)
ACCEPTED/PAID ──发货──▶ SHIPPED
SHIPPED ──签收──▶ RECEIVED
RECEIVED ──超时/确认──▶ COMPLETED
* 任意未完结态 ──取消/全额退款──▶ CLOSED
* PAID 后 ──申请售后──▶ (挂起, 主单置 AFTERSALE)
```

### 8.3 售后单状态机
```
CREATED(申请) → MERCHANT_REVIEW(审核中)
   → APPROVED(通过) → REFUNDING(退款中) → REFUNDED(已退款) → CLOSED
   → REJECTED(拒绝) → BUYER_ARBITRATE(仲裁中) → PLATFORM_DECISION → 执行 → CLOSED
   → CANCELLED(买家撤销)
退货场景: APPROVED → WAIT_BUYER_RETURN(待退货) → WAIT_SELLER_CONFIRM(待验货) → REFUNDING
```

### 8.4 状态流转约束（技术要点）
- 所有流转须经状态机引擎校验，禁止越级（如 SHIPPED 不能直接到 PAID）。
- 每次流转写 OrderLog：`{from, to, reason, operator, role, ts}`。
- 并发安全：状态字段加乐观锁/版本号，冲突重试。
- 事件驱动：状态变更发布领域事件（OrderPaid / Shipped / Completed），下游（积分、营销、风控、通知）订阅。

## 九、非功能需求（Non-Functional Requirements）

| 类别 | 要求 |
| --- | --- |
| 性能 | 下单 P99 ≤ 300ms；列表查询 P95 ≤ 800ms；大促峰值 ≥ 5万单/分钟可水平扩展 |
| 可用性 | 订单核心服务 SLA ≥ 99.95%；支付/履约关键链路多活 |
| 一致性 | 库存预占最终一致；支付以渠道回调+查单为准；状态机强一致 |
| 安全 | 全链路 HTTPS；敏感字段加密（手机号/地址脱敏）；权限 RBAC+行级；操作审计 |
| 合规 | 7天无理由、消保、个人信息保护法（PIPL）、支付合规、发票合规 |
| 可观测 | 全链路 TraceId；关键指标监控告警；订单流水可追溯 |
| 可扩展 | 状态机/分账/超时规则可配置；预留跨境/O2O 扩展点 |
| 幂等 | 支付回调、发货回传、退款均幂等，防重复处理 |

## 十、关键数据模型（核心字段）

`order_id, buyer_id, total_amount, discount_amount, pay_amount, coupon_id, status, create_time, pay_time, close_time, close_reason, invoice_type, remark`

`sub_order_id, order_id, merchant_id, sku_list, item_amount, ship_status, logistics_no, warehouse_id, receive_time`

`pay_id, order_id, channel, pay_amount, status(INIT/PAID/REFUND/CLOSED), channel_trade_no, pay_time`

`aftersale_id, order_id, sub_order_id, type(REFUND/RETURN/EXCHANGE), status, refund_amount, reason, evidence[], logistics_no`

`id, order_id, from_status, to_status, reason, operator_id, role, create_time`

## 十一、外部系统对接（Integrations）

| 系统 | 对接方式 | 说明 |
| --- | --- | --- |
| 支付网关 | API + 异步回调 | 微信/支付宝/余额，回调验签 |
| 商品/库存 | RPC/事件 | 下单校验、库存预占与回滚 |
| 物流轨迹 | 订阅推送 | 快递鸟/菜鸟，轨迹回传 |
| 营销/优惠 | 同步计算 | 优惠券/积分，结果消费 |
| 消息中心 | MQ | 通知事件订阅 |
| 财务/结算 | 文件+API | 对账文件、分账、发票 |
| 风控引擎 | 同步+异步 | 下单/退款实时决策 |

## 十三、权限与安全（Security）

- 认证：买家/商家/后台统一 SSO + 多因子（商家/财务）。
- 授权：RBAC + 数据行级（商家仅本店、财务脱敏）。
- 审计：所有写操作（发货/退款/仲裁/配置）留痕可追溯。
- 数据安全：手机号/地址加密存储与展示脱敏；导出带水印与权限审批。
- 接口安全：签名 + 限流 + 防重放；支付回调验签。

## 十五、风险与依赖（Risks & Dependencies）

| 风险/依赖 | 影响 | 缓解 |
| --- | --- | --- |
| 支付渠道回调不稳定 | 掉单/重复支付 | 回调+主动查单+幂等 |
| 大促峰值流量 | 下单/支付雪崩 | 限流、削峰、扩容预案 |
| 库存超卖 | 资损/客诉 | 预占+锁+补偿 |
| 物流回传延迟 | 状态不更新 | 兜底定时拉取 |
| 三方对账差异 | 结算差错 | 自动差异标记+复核 |
| 历史订单迁移 | 数据不一致 | 双写+校验+回滚方案 |

## 十六、验收标准（Definition of Done）

1. 下单→支付→发货→签收→完成主流程端到端跑通，状态一致。
2. 三种售后类型全流程跑通，退款金额校验正确、原路返回。
3. 超时关单、自动确认收货、自动同意退款（超时）规则生效。
4. 商家/运营/客服后台核心操作可用，权限与脱敏正确。
5. 支付回调掉单场景有查单补偿，无资损。
6. 大促压测达到 ≥ 5万单/分钟，P99 下单 ≤ 300ms。
7. 对账差错率 ≤ 0.1%，分账结算正确。
8. 关键操作审计日志完整，可追溯。
9. 合规项（7天无理由、PIPL、发票）满足。
